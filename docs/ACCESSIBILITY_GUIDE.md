# Accessibility Guide: PaletteOS

## Purpose
While the `ACCESSIBILITY_ENGINE.md` dictates the technical implementation of accessibility checks for user-generated palettes, *this* document dictates the accessibility requirements for the PaletteOS application interface itself.

## Responsibilities
- Ensure PaletteOS is usable by keyboard-only users.
- Ensure PaletteOS is fully readable by screen readers (NVDA, VoiceOver, JAWS).
- Provide a robust standard for ARIA attributes and focus management.

## Guidelines

### 1. Keyboard Navigation
- Every interactive element (sliders, buttons, color inputs) must be reachable via the `Tab` key.
- The `Focus` state must be extremely clear. We will use a highly visible `ring` (e.g., `ring-2 ring-blue-500 ring-offset-2`) on all focused elements.
- Custom components (like a draggable color swatch) must map to standard keyboard controls (Arrow keys to move/adjust, Enter to select).

### 2. Screen Reader Compatibility
- **Color Representation**: A hex code like `#3B82F6` is meaningless to a screen reader. All colors must have a visually hidden semantic label generated by the engine (e.g., `aria-label="Color: Electric Blue, Hex 3B82F6"`).
- **Live Regions**: When a user generates a new palette, a visually hidden `aria-live="polite"` region must announce: "New palette generated with 5 colors."

### 3. Motion & Animation
- As detailed in `ANIMATION_GUIDE.md`, all UI animations must respect the `prefers-reduced-motion` media query.

### 4. Semantic HTML
- Never use a `<div>` with an `onClick` handler when a `<button>` is appropriate.
- Ensure proper heading hierarchy (H1 -> H2 -> H3) on all pages.

## Best Practices
- **Testing**: Developers must test their UI components using the keyboard *only* before opening a Pull Request.
- **Skill Usage**: Rely on the `accessibility-compliance-accessibility-audit` skills for deeper insights into aria-patterns.

## Risks
- Custom UI controls (like the 2D color picker matrix) are notoriously difficult to make accessible.
  - *Mitigation*: Fall back to standard `<input type="color">` natively, and overlay our custom UI visually, keeping the native input accessible to the DOM.

## Developer Notes
- "Accessibility is not a feature; it is a baseline." - This applies heavily to a tool that purports to grade other people's accessibility.
